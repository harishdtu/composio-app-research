AI Product Ops — 100-app take-home

88 of 100 apps offer OAuth2 or an API key as an auth option. The interesting part is which 12 don't work that way at all, and why.

Before Composio builds a toolkit for an app, someone has to figure out its auth, whether it's self-serve or gated, and whether it can be agent-callable at all. This is that research, run across 100 real apps by an agent instead of by hand — clustered into patterns, verified against live docs, and shown with the misses left in.

01 — The patterns

The headline, stated plainly

Two auth flows cover almost everything. Self-serve is the default, not the exception — but it's wildly uneven by category. And the blocker that shows up most isn't payment, it's the absence of a developer story at all.

Primary auth method (100 apps)

How this is counted: "primary" = the first auth method listed per app in the dataset (i.e. the one a developer would reach for first). Every number here is computed straight from apps_100.json — see analyze.py in the repo.

    03 — The agent

    What's automated, and where a human was needed

    Two separate LLM passes — a researcher and an adversarial verifier — plus one deterministic clustering script. Everything below is real, runnable code, not pseudocode. See the repo in Section 04 to run it yourself.

    1
    Research pass agent
    research_agent.py — one Gemini call per app with Google Search grounding turned on, forced into a strict JSON schema, instructed to cite the app's real docs and say "blocked / low confidence" instead of guessing when it can't find anything public.
    2
    Verification pass agent, adversarial
    verify_agent.py — a separate prompt re-checks a sample and actively tries to disprove pass 1 against live docs, rather than just re-asking the same question and calling agreement "proof."
    3
    Pattern analysis script, deterministic
    analyze.py — no LLM call at all. Clusters the 100 rows into auth distribution, access-by-category, blocker taxonomy, MCP coverage, easy-wins vs needs-outreach. This is what produced the numbers in Section 01.
    H
    Human spot-check human
    Every row in the verification sample was independently re-opened and read by hand against its cited URL before being trusted — see Section 05 for exactly which ones, and which two are still unresolved.
    H
    Judgment calls human
    The agent flags ambiguity, but resolving it is a human call — e.g. deciding that iPayX's "gated behind 402 after free trial" still counts as self-serve, or that Sherlock (a local CLI tool) belongs in "blocked: not a hosted API" rather than being scored on auth it doesn't have.
    # research_agent.py — the actual system prompt sent per app "Use Google Search to find its REAL developer documentation. If you cannot find public developer documentation after searching, do NOT guess — set verdict:"blocked" and explain why. Set confidence:"low"." # run it $ export GEMINI_API_KEY=AIza... $ python3 research_agent.py # full 100-app pass $ python3 verify_agent.py --sample 15 # adversarial re-check $ python3 analyze.py > data/analytics.json

    05 — The verification

    Where the agent was wrong, and how we know

    15 of 100 apps, oversampled toward the niche/ambiguous ones, were hand-checked against live docs. Six first-pass guesses were materially wrong and got corrected. Two apps still have no public answer — and stay flagged low-confidence rather than filled in.

    60%→100%
    60% = what an LLM's training-knowledge-only guess got right on this sample, unaided. 100% = after the live-docs verification loop corrected every wrong or incomplete guess. The number that actually matters is the correction rate: 6 of 15 (40%) needed a real fix — that's the loop earning its keep, not a claim that the full 100-row dataset is 100% hand-verified.
